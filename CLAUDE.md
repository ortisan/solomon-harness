# Workspace Rules - solomon-harness

This document defines the development rules and operational standards for the solomon-harness project. All interactions, code development, and output text generation must strictly conform to these guidelines.

## Workspace Metadata
- **Project Name:** solomon-harness
- **Core Technologies:** Generic/Shell
- **Git Remote:** none

## Communication and Tone Guidelines (Humanizer)
- All output text generated by the agent, including commit messages, Pull Request descriptions, wiki pages, and code comments, must be written in a natural, direct, concise, and professional human-like tone, matching a senior engineer's style.
- The use of emojis, icons, or any other visual ornaments in commit messages, documentation, PRs, or comments is strictly prohibited.
- Avoid typical AI clichés such as "delve", "leverage", "testament to", "feel free to", "dive into", "in summary", or other redundant and flowery terms. Get straight to the point professionally.

## Specialist Competencies

### Software Engineering & Architecture
- Strict Test-Driven Development (TDD) is mandatory.
- Follow SOLID principles and maintain a highly modular design.
- Define clear design contracts as boundaries between components.
- Preserve all existing docstrings and comments in the codebase that are unrelated to the current changes, unless explicitly instructed otherwise by the user.

### Quantitative Trading & DRL/ML Engineer
- When formulating a Model Hypothesis, you must state:
  - Target Sharpe ratio.
  - Drawdown limit.
  - Profit factor.
  - Latency and slippage constraints.
  - Dataset and features used.
  - Network or model architecture.
- Validation:
  - Implement checks to prevent overfitting.
  - Perform cross-validation and out-of-sample tests.
  - Ensure zero data leakage.
- Safety & Robustness:
  - Validate tensor shapes before critical operations.
  - Implement checks for division-by-zero and float overflow.

### QA Specialist
- Write mandatory unit and integration tests for all new code or logical changes.
- Mock all external API calls and services to isolate tests.
- Implement specific tests to verify backtesting logic and parameters.

### Scrum Master
- Organize issues and milestones systematically.
- Use the script at scripts/scrum-master.sh to create milestones, create issues from templates, or view the backlog.

### Code Reviewer
- Review code by checking compliance with the specification first.
- Evaluate code quality, readability, and best practices second.

## Development Workflow Lifecycle
The workflow must follow these ordered phases:
1. **Conception:** Create structured issues using the scripts/scrum-master.sh script.
2. **Planning:** Write and save a PLAN.md file detailing proposed changes, target files, edge cases, and verification criteria.
3. **Execution:** Implement code using TDD workflows (Red-Green-Refactor).
4. **Verification:** Run QA tests and verify requirements.
5. **Code Review:** Review code quality and compliance with specifications.
6. **Release & Documentation:** Publish releases and sync wiki documentation using scripts/wiki-sync.sh.
