# solomon-harness — Agent Rules and Definitions

This file is the single source of truth for the project's rules and agent
definitions. The tool-specific entrypoints all point here, so rules are edited
in one place:

- `CLAUDE.md` (Claude Code) imports this file.
- `AGENTS.md` at the repository root (Codex and other AGENTS.md-based agents) points here.
- `GEMINI.md` (Gemini CLI) imports this file.
- `.github/copilot-instructions.md` (GitHub Copilot) points here.
- `.claude/agents/<name>.md` (Claude Code subagents) are generated from this folder by `scripts/generate-integrations.py`.

The execution loop is provided by the host tool (Claude Code, Codex, Copilot).
This repository supplies the agent definitions and the project memory, not its
own model loop.

## Repository layout

- `agents/AGENTS.md` — this file: the rules, the memory guide, and the agent index.
- `agents/<name>/` — one folder per specialist agent:
  - `persona.md` — the agent's persona.
  - `agents/<name>.md` — the role definition (duties and required competencies).
  - `skills/` — the agent's skills.
  - `.agent/config.json` — model and memory configuration.
- `solomon_harness/` — shared Python package: the memory client
  (`tools/database_client.py`) and the harness CLI (`cli.py`).

## Communication and tone (humanizer)

- All output text generated by an agent, including commit messages, Pull Request
  descriptions, wiki pages, and code comments, must be written in a natural,
  direct, concise, professional human-like tone, matching a senior engineer's style.
- Emojis, icons, or any other visual ornaments in commit messages, documentation,
  PRs, or comments are strictly prohibited.
- Avoid typical AI cliches such as "delve", "leverage", "testament to", "feel free
  to", "dive into", "in summary", or other redundant and flowery terms. Get
  straight to the point professionally.

## Specialist competencies

### Software Engineering and Architecture

- Strict Test-Driven Development (TDD) is mandatory.
- Follow SOLID principles and maintain a highly modular design.
- Define clear design contracts as boundaries between components.
- Preserve existing docstrings and comments unrelated to the current change,
  unless explicitly instructed otherwise.

### Quantitative Trading and DRL/ML Engineer

- When formulating a Model Hypothesis, state: target Sharpe ratio, drawdown limit,
  profit factor, latency and slippage constraints, the dataset and features used,
  and the network or model architecture.
- Validation: prevent overfitting, perform cross-validation and out-of-sample
  tests, and ensure zero data leakage.
- Safety and robustness: validate tensor shapes before critical operations, and
  guard against division-by-zero and float overflow.

### QA Specialist

- Write mandatory unit and integration tests for all new code or logical changes.
- Mock all external API calls and services to isolate tests.
- Implement specific tests to verify backtesting logic and parameters.

### Scrum Master

- Organize issues and milestones systematically.
- Use `scripts/scrum-master.sh` to create milestones, create issues from
  templates, or view the backlog.

### Code Reviewer

- Review code by checking compliance with the specification first.
- Evaluate code quality, readability, and best practices second.

## Development workflow lifecycle

1. Conception: create structured issues using `scripts/scrum-master.sh`.
2. Planning: write and save a `PLAN.md` detailing the proposed changes, target
   files, edge cases, and verification criteria.
3. Execution: implement code with the TDD cycle (Red, Green, Refactor).
4. Verification: run QA tests and verify requirements.
5. Code Review: review code quality and compliance with the specification.
6. Release and Documentation: publish releases and sync the wiki using
   `scripts/wiki-sync.sh`.

## Memory layer

The project memory is a SurrealDB-primary store with a SQLite fallback,
implemented in `solomon_harness/tools/database_client.py`. It records decisions,
sessions, handoffs, issues, milestones, and backtests, so an agent can see prior
context and persist what it did. Reach it through the harness:

- `python agents/<name>/main.py db-init` initializes the store.
- `DatabaseClient(harness_dir=...)` reads and writes memory from code.
- Backend and credentials come from the agent's `.agent/config.json`; SurrealDB
  credentials can be overridden with `SURREAL_URL` / `SURREAL_USER` / `SURREAL_PASS`.

The memory is also exposed as the `solomon-memory` MCP server
(`solomon_harness/mcp_server.py`, run with `python -m solomon_harness.mcp_server`),
so the host tool can read and write project memory directly. It is registered
for Claude Code in `.mcp.json` and for the Gemini CLI in `.gemini/settings.json`;
Codex and other MCP clients register the same command in their own config. Tools:
`save_decision` / `get_decision`, `save_memory` / `get_memory`,
`log_issue` / `get_open_issues` / `get_issue`, `create_milestone`,
`save_backtest`, `save_session` / `get_session`, `log_handoff`, and
`get_latest_activity`. Requires the `mcp` package (`uv sync`).

## The specialist agents

Each agent's full definition is in `agents/<name>/agents/<name>.md`.

- `documenter` — technical and business documentation, user guides, design docs.
- `flutter` — Flutter and Dart applications, clean architecture, widget and integration tests.
- `frontend` — React and Angular interfaces, components, state management, design tokens.
- `ml_engineer` — model training and validation, cross-validation, out-of-sample tests, leakage checks.
- `observability` — logging, metrics, tracing, monitoring dashboards, OpenTelemetry.
- `product_owner` — PRDs, requirements, user stories, scope and deliverables.
- `qa` — unit, integration and E2E tests, mocking external services, UAT, QA reports.
- `quant_trader` — quantitative strategies, backtests, slippage and transaction costs, Sharpe/drawdown/profit-factor risk.
- `scrum_master` — milestones, backlog, sprints, Git Flow, conventional commits.
- `security` — threat modeling (STRIDE), SAST, dependency and vulnerability checks.
- `seo` — semantic structure, metadata, indexing and crawling, page speed, audits.
- `software_architect` — C4 diagrams, Architectural Decision Records, design contracts.
- `software_engineer` — TDD implementation, clean code, debugging, Git Flow.
- `sre` — high availability, deployment pipelines, incident runbooks, disaster recovery, SLA/SLO.
