# Dependency and Supply-Chain Security

The standard for what this project depends on and how those dependencies are resolved, verified, scanned, and updated — covering lockfile discipline, advisory scanning, SBOMs, name-confusion attacks, SLSA provenance, and GitHub Actions hardening. The stance: every artifact that executes in dev, CI, or production is either pinned and verified or it is an accepted risk with an owner.

## Lockfile discipline

Applications deploy from a lockfile; libraries declare compatible ranges. This repo is an application-plus-CLI, so `uv.lock` is the source of truth: it pins every direct and transitive package to an exact version with SHA-256 hashes, and uv verifies those hashes at install time. Rules:

- `uv.lock` is committed and CI installs with `uv sync --locked` so a drifted lockfile fails the build instead of silently re-resolving. (The current `ci.yml` runs plain `uv sync --group dev`; tightening it to `--locked` is the standing recommendation.)
- Version ranges in `pyproject.toml` express compatibility intent (`>=2.1,<3`); reproducibility comes from the lock, not from hand-pinning `==` everywhere in `pyproject.toml`.
- If a `requirements.txt` export is ever needed, generate it with hashes and install with `pip install --require-hashes -r requirements.txt`.
- Never install from a URL, a git ref without a full commit SHA, or an unpinned `--pre` channel.

## Scanning and update flow

New CVEs appear against unchanged code, so scanning must run on a schedule, not only on PRs:

- `pip-audit` (PyPA advisory DB + OSV, no account needed) on every PR and a daily cron.
- `osv-scanner` reads `uv.lock` directly and cross-checks OSV across ecosystems — a good second source; `trivy fs` or `grype` covers container images if any ship.
- Findings inherit the remediation SLAs from the SLA skill; a Critical advisory on a reachable package is a 48-hour clock, not a Dependabot ticket to age.
- Dependabot or Renovate raises update PRs; nothing auto-merges without the full test suite and scanners green. Configure a cooldown (Renovate `minimumReleaseAge: "7 days"`, or Dependabot's cooldown setting) so a freshly compromised release is not pulled the hour it lands — most hijacked versions are yanked within days.

## SBOM

Publish a CycloneDX SBOM (spec is ECMA-424; current version 1.6) per release: `uvx cyclonedx-py environment` or `cyclonedx-py requirements` against the lock export, attached as a release artifact. The SBOM is what lets a consumer answer "did we ship the affected version?" during the next log4shell-style event without re-deriving the tree. SPDX is the acceptable alternative when a consumer requires it.

## Typosquatting and dependency confusion

Name-confusion attacks are routine on PyPI. Two canonical cases: `ctx` (2022, hijacked package exfiltrating AWS keys) and the PyTorch-nightly `torchtriton` dependency-confusion compromise (December 2022), where a public package shadowed a private index name. Defenses:

- Review every new dependency name character-by-character before adding it; prefer packages with history, maintainers, and provenance.
- Never mix `--extra-index-url` with an internal index; resolution order is undefined for pip and an attacker who registers the internal name publicly wins. Use a single index, or uv's `[[tool.uv.index]]` with explicit per-package pinning.
- The install cooldown above also blunts hijacks of legitimate packages (the `xz-utils` backdoor, CVE-2024-3094, sat in releases for weeks before detection — upstream compromise is not hypothetical).

## SLSA provenance

SLSA v1.0 Build track levels, practically: L1 = the build produces provenance; L2 = provenance is signed and generated by the hosted build platform; L3 = the build runs isolated so the provenance is unforgeable by the build itself. On GitHub, `actions/attest-build-provenance` gets release artifacts to L2 with a few lines; the `slsa-github-generator` reusable workflow reaches L3. Verify consumed attestations with `gh attestation verify` or `slsa-verifier`. For this project the bar is: release artifacts carry signed provenance, and any binary tool CI downloads is either checksum-verified or attestation-verified.

## GitHub Actions hardening

CI is a supply-chain target with write access to the repo. The `tj-actions/changed-files` compromise (March 2025, CVE-2025-30066) rewrote existing version tags to point at malicious code and dumped runner secrets from thousands of repos — mutable tags are the attack surface. This repo already does the main control correctly: every action in `.github/workflows/ci.yml` is pinned to a full commit SHA with a version comment, e.g. `actions/checkout@692973e3d937129bcbf40652eb9f2f61becf3332 # v4.1.7`. Keep it that way, and:

- Top-level `permissions: contents: read` (already set in `ci.yml`); grant elevated scopes per job only.
- Never combine `pull_request_target` with a checkout of the PR head; that hands fork code the base repo's secrets.
- Do not build cache keys or run steps from attacker-influenced strings (branch names, PR titles) — script injection via `${{ }}` interpolation is the other classic. Pass untrusted values through `env:` and quote them.
- Audit workflows with `zizmor` when adding or changing one; let Dependabot's `github-actions` ecosystem keep SHA pins updated with reviewable diffs.

## Licensing

Block copyleft licenses incompatible with the project license (GPL/AGPL) and any package with no license metadata; keep an explicit allowlist and check it in the same scheduled job.

## Common pitfalls

- Pinning direct dependencies but resolving transitives fresh at install time; the lockfile must cover the whole tree with hashes or it covers nothing.
- Scanning only on PRs, so a CVE published against the untouched `main` waits weeks for someone to open one.
- Auto-merging dependency updates without a cooldown, importing a hijacked release hours after it ships.
- Pinning an action to a tag (`@v4`) and calling it pinned; tags are mutable and were exactly the vector in CVE-2025-30066.
- `--extra-index-url` alongside a private index, reopening dependency confusion.
- Generating an SBOM from `pyproject.toml` ranges instead of the resolved environment, so it lists what was asked for, not what shipped.
- Treating a scanner "no findings" as safety; scanners only know published advisories — compromise-in-progress (xz-style) is caught by provenance and cooldowns, not CVE feeds.

## Definition of done

- [ ] `uv.lock` committed; CI installs with `--locked`/`--frozen` semantics so resolution drift fails the build.
- [ ] `pip-audit` plus a second source (osv-scanner or trivy/grype) run on every PR and on a daily schedule; findings enter the SLA tracker.
- [ ] Update automation has a release cooldown and never auto-merges without the full gate green.
- [ ] CycloneDX SBOM generated from the resolved environment and attached to each release.
- [ ] No mixed indexes; every new dependency name reviewed and its maintenance signals checked.
- [ ] Release artifacts carry signed SLSA provenance (L2 minimum); downloaded build tools are checksum- or attestation-verified.
- [ ] Every GitHub Action pinned to a full commit SHA with a version comment; workflow permissions least-privilege; no `pull_request_target` + PR-head checkout; untrusted values only via `env:`.
- [ ] License allowlist enforced in CI.
