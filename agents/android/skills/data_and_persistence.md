# Data and Persistence

Persist app data so the database is the single source of truth, the UI observes it reactively through Flow, and the network is treated as a fallible writer that fills the cache rather than a thing the UI talks to directly. Make Room the local store of record, DataStore the place for small key-value and typed settings, WorkManager the executor for deferrable durable work, and Retrofit/OkHttp the typed network edge, and wire them into an offline-first repository that survives process death and airplane mode.

## Room: entities, DAOs, Flow, KSP

Use Room 2.7+ (2.8 on the stable channel by 2026, with built-in Kotlin Multiplatform and a coroutine-native driver). Generate code with KSP2 (`com.google.devtools.ksp`), never the legacy kapt path; kapt forces a full Java stub round-trip and roughly doubles annotation-processing time on a Room-heavy module.

```kotlin
// build.gradle.kts (version catalog refs omitted for brevity)
plugins { id("com.google.devtools.ksp") }
dependencies {
    implementation("androidx.room:room-runtime:2.7.1")
    implementation("androidx.room:room-ktx:2.7.1")   // suspend + Flow DAO support
    ksp("androidx.room:room-compiler:2.7.1")
}
room { schemaDirectory("$projectDir/schemas") }      // commit these JSON files
```

```kotlin
@Entity(
    tableName = "account",
    indices = [Index(value = ["remote_id"], unique = true)],
)
data class AccountEntity(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    @ColumnInfo(name = "remote_id") val remoteId: String,
    val balanceMinor: Long,            // store money as integer minor units, never Double/float
    val updatedAt: Long,               // epoch millis, for staleness checks
)

@Dao
interface AccountDao {
    @Query("SELECT * FROM account ORDER BY updatedAt DESC")
    fun observeAll(): Flow<List<AccountEntity>>      // cold Flow, re-emits on every write

    @Upsert
    suspend fun upsert(items: List<AccountEntity>)    // insert-or-update, atomic

    @Transaction
    suspend fun replaceAll(items: List<AccountEntity>) {  // atomic refresh
        clear(); upsert(items)
    }

    @Query("DELETE FROM account") suspend fun clear()
}
```

Rules that matter:

- Return `Flow<T>` for reads the UI observes; Room re-runs the query and emits on any write to the touched tables, which is what makes the DB the single source of truth. Use `suspend` for one-shot writes. Never expose a blocking `LiveData`-only DAO from new code.
- Collect Flows with `.flowOn(Dispatchers.IO)` is unnecessary — Room already dispatches queries on its own executor. Do not wrap DAO calls in `withContext(IO)`; it is redundant and hides the suspension.
- Annotate any DAO method that returns an `@Relation` or joins child tables with `@Transaction`, or you can read a torn snapshot taken across two internal queries.
- Use `@Upsert` (Room 2.5+) instead of `@Insert(onConflict = REPLACE)`. REPLACE deletes then inserts, which fires foreign-key `ON DELETE CASCADE` and resets autogenerated PKs; upsert updates in place.

## Room: schema migrations

Set `exportSchema = true` and commit the generated `schemas/<db>/<version>.json`. Auto-migrations diff those files; without them you cannot use `@AutoMigration` and you lose the CI guard that catches an un-migrated schema change.

```kotlin
@Database(
    entities = [AccountEntity::class],
    version = 3,
    exportSchema = true,
    autoMigrations = [AutoMigration(from = 2, to = 3, spec = Migrate2to3::class)],
)
abstract class AppDatabase : RoomDatabase()

@RenameColumn(tableName = "account", fromColumnName = "amount", toColumnName = "balanceMinor")
class Migrate2to3 : AutoMigrationSpec
```

- Prefer `@AutoMigration` for column add/rename/delete and table renames. Drop to a manual `Migration(2, 3)` with explicit SQL only for data transforms auto-migration cannot express (splitting a column, back-filling).
- Never ship `fallbackToDestructiveMigration()` in a release build that holds user data; it silently drops every table on a version bump and is a data-loss incident waiting to happen. It is acceptable only behind a debug flag.
- Add a `MigrationTestHelper` instrumentation test for every migration: create the DB at version N, run the migration, assert the data and schema at N+1. A migration with no test is unverified.

## DataStore: Preferences and Proto

Use DataStore (`androidx.datastore` 1.1+) for everything you would have put in `SharedPreferences`. SharedPreferences blocks on the UI thread via `apply()`/`commit()` and offers no transactional consistency; DataStore is fully async over a single Flow and applies updates atomically. Never run both against the same data.

Preferences DataStore for loose, untyped keys:

```kotlin
val Context.settings by preferencesDataStore(name = "settings")
val THEME = stringPreferencesKey("theme")

val theme: Flow<String> = context.settings.data
    .catch { e -> if (e is IOException) emit(emptyPreferences()) else throw e }
    .map { it[THEME] ?: "system" }

suspend fun setTheme(v: String) { context.settings.edit { it[THEME] = v } }
```

Proto DataStore for typed, structured settings — define a schema and get compile-time safety plus a real default:

- Always wire the `.catch { }` that re-emits `emptyPreferences()` (or the proto default) on `IOException`; a corrupt file otherwise crashes every reader. Provide a `corruptionHandler` for proto stores.
- Create exactly one DataStore instance per file for the process lifetime (the `by preferencesDataStore` delegate enforces this). A second instance on the same file throws.
- DataStore is for small data (single-digit KB). Do not store lists that grow unbounded or anything you query by field — that is Room's job. Reads deserialize the whole file.
- Do not put auth tokens or secrets in plain DataStore. Back sensitive values with the AndroidKeyStore / `EncryptedFile` and store only the ciphertext.

## WorkManager: constraints, backoff, expedited, unique work

Use WorkManager 2.10+ for deferrable work that must survive process death and reboot (sync, upload, cleanup). Do not use it for work that must run now while the user watches — that is a coroutine on a lifecycle scope. WorkManager guarantees execution, not immediacy.

```kotlin
val sync = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(
        Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .setRequiresBatteryNotLow(true)
            .build()
    )
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS) // min 10s, capped 5h
    .setExpedited(OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST)
    .build()

WorkManager.getInstance(ctx).enqueueUniqueWork(
    "account-sync", ExistingWorkPolicy.KEEP, sync,    // dedupe by name
)
```

```kotlin
class SyncWorker(ctx: Context, p: WorkerParameters) : CoroutineWorker(ctx, p) {
    override suspend fun doWork(): Result =
        when (repository.refresh()) {
            is Ok -> Result.success()
            is Transient -> Result.retry()       // honors backoff; do not loop forever
            is Permanent -> Result.failure()     // bad request, auth — retry will never help
        }
}
```

- Use `CoroutineWorker` for suspend work; it runs on `Dispatchers.Default` and cancels cooperatively when the OS stops the job. Return `Result.retry()` only for transient failures; returning it for a 400/permanent error burns battery on a doomed exponential backoff.
- `enqueueUniqueWork` / `enqueueUniquePeriodicWork` with a stable name is the deduplication primitive. Pick `KEEP` (ignore the new request if one is queued), `REPLACE`, or `APPEND_OR_REPLACE` deliberately; the default of enqueuing afresh each app start stacks duplicate workers.
- Expedited work (`setExpedited`) runs quickly but is quota-limited per app; always pass `OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST` so it degrades to a normal job instead of throwing when the foreground quota is exhausted. On API 31+ it maps to an expedited job; on older APIs to a foreground service, which needs `setForeground`.
- Minimum periodic interval is 15 minutes; anything shorter is clamped. For tighter cadence you do not control WorkManager — reconsider the requirement.
- Inject dependencies with Hilt's `@HiltWorker` + `HiltWorkerFactory`; do not reach into a global singleton from inside a worker.

## Retrofit / OkHttp: interceptors, converters, timeouts, errors

Use Retrofit 3.x on OkHttp 5.x with `suspend fun` endpoints and the kotlinx.serialization converter (faster and Kotlin-aware versus Gson/Moshi reflection paths).

```kotlin
val client = OkHttpClient.Builder()
    .callTimeout(30, TimeUnit.SECONDS)        // whole-call ceiling, the important one
    .connectTimeout(10, TimeUnit.SECONDS)
    .readTimeout(15, TimeUnit.SECONDS)
    .retryOnConnectionFailure(true)
    .addInterceptor(AuthInterceptor(tokenStore))     // app layer: add headers
    .addNetworkInterceptor(loggingInterceptor)       // network layer: see real wire bytes
    .build()

val retrofit = Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .client(client)
    .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
    .build()

interface AccountApi {
    @GET("v1/accounts") suspend fun accounts(): List<AccountDto>   // throws on non-2xx
}
```

- Set `callTimeout` as the absolute upper bound; `connect`/`read`/`write` timeouts alone do not cap a slow-trickle response. Without `callTimeout` a stalled stream can hang a coroutine indefinitely.
- Distinguish interceptor layers: application interceptors (`addInterceptor`) run once and see the logical request — put auth, retry, and header logic here; network interceptors (`addNetworkInterceptor`) run per physical request and see redirects and cached responses — put wire logging and compression here.
- Map errors explicitly. A `suspend` Retrofit call throws `HttpException` on non-2xx and `IOException` on transport failure. Wrap every call in a `runCatching` that converts these into a sealed `NetworkResult` (`Success`/`HttpError(code)`/`NetworkError`/`Unknown`); never let a raw `HttpException` reach the ViewModel.
- Refresh tokens with an OkHttp `Authenticator`, not an interceptor — the Authenticator is invoked exactly on a 401 and can synchronize a single refresh; doing it in an interceptor races N concurrent requests into N refreshes.
- Add `ChuckerInterceptor` (debug only) for on-device traffic inspection; never include the logging/Chucker interceptors at `BODY` level in release — they leak tokens and PII into logcat.

## Offline-first: single source of truth

The repository exposes Room as the only thing the UI observes and refreshes it from the network out of band. The UI works fully offline and updates the instant a write lands.

```kotlin
fun accounts(): Flow<List<Account>> = dao.observeAll().map { it.toDomain() }

suspend fun refresh(): SyncResult = networkResult {
    val remote = api.accounts()                 // network
    dao.replaceAll(remote.toEntities())         // write-through; Flow above re-emits
}.also { result -> if (result.isError) scheduleRetryWork() }
```

- One read path: the UI collects the DB Flow; it must not also read the network and merge in the ViewModel. Network responses only ever flow into the DB, and the DB is the only emitter the UI sees.
- Define a staleness policy with `updatedAt`: serve cache immediately, trigger a background refresh when data is older than a threshold (for example 5 minutes for a balance, hours for static reference data). Do not block the UI on the refresh.
- For user writes, apply an optimistic local update first (write to Room, UI updates instantly), enqueue a unique WorkManager job to push it, and reconcile on success or roll back / mark the row `pendingSync` on permanent failure. This is what makes the app usable on a subway.
- Use HTTP caching (OkHttp `Cache` + `ETag`/`Cache-Control`) for the network layer as an optimization, but it is not the offline store — Room is. The two solve different problems; do not conflate them.
- Paginate large lists with Paging 3 backed by a `RemoteMediator` so the network fills Room page by page and the UI still observes Room only.

## Common pitfalls

- Wrapping suspend DAO calls in `withContext(Dispatchers.IO)`: redundant (Room dispatches itself) and signals a misunderstanding; reject it.
- `@Insert(onConflict = REPLACE)` used where `@Upsert` is meant — it cascades deletes and resets autogen PKs.
- `fallbackToDestructiveMigration()` in a release build holding user data: silent total data loss on version bump.
- A schema change with no committed `schemas/*.json` and no `MigrationTestHelper` test — the migration is unverified and CI cannot catch it.
- SharedPreferences left in place beside DataStore for the same setting, producing two diverging sources of truth.
- WorkManager `Result.retry()` returned for a permanent 4xx, looping exponential backoff against a request that can never succeed.
- Expedited work without `OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST`, which throws once the foreground quota is spent.
- Enqueuing the same logical job without a unique name, stacking duplicate workers on every app launch.
- Token refresh done in an interceptor instead of an `Authenticator`, firing N parallel refreshes under concurrency.
- No `callTimeout` on the OkHttp client, so a slow-trickle response hangs a coroutine forever.
- ViewModel reading both network and DB and merging them — two read paths break the single source of truth and produce flicker.
- BODY-level logging or Chucker shipped in release, leaking tokens and PII to logcat.

## Definition of done

- [ ] Room is the single source of truth; UI observes DAO `Flow`s only, and network responses are written through to Room before the UI sees them.
- [ ] DAOs use `Flow` for reads and `suspend` for writes; `@Transaction` guards every relation/multi-query read; `@Upsert` replaces conflict-REPLACE inserts.
- [ ] KSP2 (not kapt) generates Room code; `exportSchema = true` and `schemas/*.json` are committed.
- [ ] Every migration is an `@AutoMigration` or explicit `Migration`, covered by a `MigrationTestHelper` test; no destructive fallback in release.
- [ ] DataStore (Preferences or Proto) replaces all SharedPreferences use, reads `.catch` IOException, and holds only small data; secrets are KeyStore-backed.
- [ ] Deferrable durable work runs in a `CoroutineWorker` via `enqueueUnique*` with a stable name, correct `ExistingWorkPolicy`, exponential backoff, constraints, and expedited fallback policy.
- [ ] Workers return `retry` only for transient failures and `failure` for permanent ones; dependencies are injected via `@HiltWorker`.
- [ ] Retrofit 3 / OkHttp 5 client sets `callTimeout` plus connect/read timeouts, uses the kotlinx.serialization converter, refreshes tokens via an `Authenticator`, and maps `HttpException`/`IOException` into a sealed result before the ViewModel.
- [ ] Wire logging / Chucker are debug-only and never at BODY level in release.
- [ ] Offline-first behavior verified: list and detail render with the network off, optimistic writes apply locally and sync via WorkManager, and a staleness threshold drives background refresh.
- [ ] Tests cover DAO Flow emission on write, each migration, repository cache-then-network behavior, and network error mapping, with Retrofit/OkHttp mocked (MockWebServer) and no real network calls.
