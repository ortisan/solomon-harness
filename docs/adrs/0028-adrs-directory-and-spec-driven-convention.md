# ADR-0028: docs/adrs as the decision home, and the spec-driven issue convention

- Status: accepted
- Date: 2026-07-14
- Deciders: software_architect, product_owner (maintainer decision, #221)
- Issue: #234 (S2a of #221; also records the convention shipped by #233)

## Context and problem statement

Umbrella #221 established two durable artifact trees for every
harness-installed project: `docs/specs/` (one spec per feature issue) and a
single canonical home for decision records. The records lived under
`docs/adr/`, a name the maintainer decided to consolidate as `docs/adrs/`.
S1 (#233) shipped the spec convention with its decision record deliberately
deferred to this slice, because a record written before the rename would cite
a directory about to move — the deferral condition was tracked at #237's
review (#236 blocked on this ADR landing).

## Decision drivers

- One canonical, plural-named home so install-time wiring (S3, #236) can
  reference a stable path in every LLM instruction file.
- Zero history or content loss: 26 numbered records existed at migration time.
- Machine-checkable integrity: the move and the convention must be enforced by
  validators and tests, not prose.

## Considered options

- Rename via `git mv` with every reference repointed in the same change, and
  record both decisions (the home and the spec convention) in one ADR.
  Rejected alternative: keep `docs/adr/` and alias `docs/adrs/` — two live
  names for one tree invites drift and doubles every future reference.
- Rejected alternative: write the spec-convention ADR in S1 under the old
  path and re-home it here — churn on a record whose whole purpose is a
  stable citation.

## Decision outcome

Chosen option "rename with same-change repointing, one combined record".
`docs/adrs/` is the single canonical decision home: all records, the 0000
template, and the README moved with content unchanged; `check-adr-unique`
targets it by default; a migration gate asserts the full uniquely-numbered
set is present, the old tree is gone, and a repo-wide negative-lookahead scan
finds zero `docs/adr` references. The spec-driven issue convention (shipped
by #233) is hereby recorded: every feature/story issue generates
`docs/specs/<N>-<slug>.md` from the house template's seven sections, enforced
by `scripts/spec-lint.py` in CI and generated by `/solomon-issue` via the
Write tool.

### Consequences

- Positive: one stable path for S3's install wiring; specs and decisions are
  both first-class, validated artifact trees; the S1 deferral condition is
  discharged and #236 is unblocked.
- Negative: external links to old `docs/adr/...` URLs (wiki pages, merged PR
  bodies, closed issues) are not rewritten — history keeps the old form, and
  GitHub serves moved files only from the new path.
- Follow-ups: S2b (#235) wires automatic capture into `docs/adrs/`; S3 (#236)
  broadcasts both paths at install time.

## More information

Migration gates: `tests/test_adr_migration.py`. Spec convention definition:
`docs/specs/README.md`; validator: `scripts/spec-lint.py`; generation:
`.claude/commands/solomon-issue.md` step 8 (ADR-0025 established the
content-gate enforcement pattern both conventions use). ADR-0024 remains
reserved by open PR #212. This decision is also recorded in the project
memory via `save_decision`.
