# PLAN — #11 qa skills to canonical depth

Issue: #11 (child of epic #6) · Branch: `feature/qa-skills-canonical-depth` (base `main`)
Author: software_engineer / qa · Date: 2026-06-28

## Problem statement

`qa` carries 16 skill files, but 8 role-core ones are thin stubs (62–244 words)
against the canonical reference (`agents/auth_engineer/skills/`, 2.4–3.8k words,
and the already-deep qa exemplars `test_planning_and_traceability.md`,
`ci_quality_gates.md`, `defect_triage_and_lifecycle.md`). Shallow skills produce
shallow artifacts at the `/solomon-review` gate. Closes the qa slice of #6.

## Proposed change and the boundary it touches

Deepen the 8 thin qa skills in place to canonical depth. Pure documentation/content
change under `agents/qa/skills/`. No Python, no public contract, no data model —
the `solomon_harness` package is untouched. The only generated artifact that may
change is the agent's "Active Skills" block, regenerated by `document-skills.py`.

Canonical format every rewritten file must follow (from the in-repo exemplar
`agents/qa/skills/test_planning_and_traceability.md`):

1. `# H1 Title`.
2. A sharp first-paragraph summary (what the skill is, why it gates quality).
3. Topic `##` sections that name concrete standards/tools and carry at least one
   worked example (code, table, or config).
4. Cross-references to sibling skills in backticks — respecting each skill's
   ownership boundary so concepts are not duplicated (consolidation AC).
5. Closing `## Common pitfalls` and `## Definition of done` (checklist).
6. >= ~600 words. Must NOT restate the shared rules in `agents/AGENTS.md`.

## Per-file content contract (ownership boundaries prevent duplication)

- **test_design_rules** — owns case-level technique + the canonical boundary set
  (empty, single, max, zero, negative, None, NaN, inf/overflow, unicode, timezone/DST).
  Equivalence partitioning, BVA, decision tables, state-transition, pairwise
  (`allpairspy`/PICT), property-based (`hypothesis` `@given` worked example with an
  invariant + shrinking). The planning skill decides *which* criteria warrant each;
  this skill is the *how*.
- **the_test_pyramid_target_distribution** — owns the level mix. Cohn pyramid vs
  ice-cream-cone/hourglass anti-patterns, Google small/medium/large sizing, the
  ~70/20/10 heuristic (not a quota), per-level speed/determinism budget, detecting
  an inverted pyramid by marker count + timing. Worked example: `pytest` markers +
  a CI timing breakdown.
- **coverage_a_floor_not_a_finish_line** — owns coverage measurement. Line vs branch
  vs path, `coverage.py`/`pytest-cov` config (`--cov-branch`, `fail_under`,
  `exclude_lines`, `# pragma: no cover` discipline), why coverage ≠ good tests
  (assertion-free execution), diff/patch coverage on PRs (`diff-cover`), mutation
  score as the real signal (defers to `mutation_testing`). Worked example: a
  `[tool.coverage]` block + a branch-coverage gap.
- **flaky_tests** — owns flaky policy. Root-cause taxonomy (async/timing, order &
  shared state, nondeterministic time/random/data, external deps, resource leaks),
  detection (`pytest-randomly` for order, reruns), a **quarantine policy** with an
  owner + SLA (not a silent skip), deflaking via pinned seed/clock
  (`freezegun`/`time-machine`). Worked example: quarantine mark + the fix.
- **uat** — owns acceptance & sign-off. UAT (validation) vs system test
  (verification), cases derived from acceptance criteria/user stories (BDD/Gherkin),
  production-like anonymized data (never mocks), the sign-off gate + roles, severity
  scale (Blocker/Critical/Major/Minor/Trivial) and which gate release, alpha/beta
  entry-exit criteria. Worked example: a UAT case table + a Gherkin scenario tied to AC-ID.
- **qa_report_the_required_output** — owns the gate artifact `/solomon-review`
  consumes. Required structure: scope/build, requirements-coverage + code-coverage
  numbers, results by risk band, defect list with severity + IDs, flaky/quarantine
  status, mutation score, explicit pass/fail verdict mapped to exit criteria, evidence
  (run IDs/logs), and the go/no-go rule. Worked example: a filled report skeleton.
- **mocking_and_isolation_mock_all_external_services** — owns test doubles + boundary
  isolation. Meszaros taxonomy (dummy/stub/spy/mock/fake), mock-at-the-boundary-you-own
  vs don't-mock-what-you-don't-own, tools (`unittest.mock`/`monkeypatch`, `responses`/
  `respx`, `moto`, in-memory fakes), consumer-driven contract tests (`pact`) to catch
  mock drift, deterministic clock/uuid/random seams, the over-mocking smell. Worked
  example: `respx` mocking an `httpx` call + a contract sketch.
- **backtesting_verification_specific_because_finance_bugs_are_silent** — owns the
  qa↔quant_trader seam. Why finance bugs are silent, look-ahead/peeking bias (shift/lag
  assertions, point-in-time data), train/test leakage, survivorship bias, slippage +
  transaction-cost + funding modeling (assert applied & non-zero), walk-forward /
  out-of-sample, reproducibility (pinned seed + data snapshot + clock), invariants
  (PnL identity, position limits, Sharpe/drawdown recompute cross-check). Worked
  example: a `pytest` asserting no look-ahead + a cost-applied assertion.

## Edge cases as observable outcomes

- A rewrite that pads to the word floor without naming concrete tools/standards
  fails review and is rewritten — depth, not length.
- Two files that both explain BVA → duplication; the boundary set lives only in
  `test_design_rules`, others cross-reference it.
- A cross-reference to a renamed/absent sibling file → broken link; the consolidate
  pass checks every backticked skill name resolves to a real file.
- `document-skills.py` mutating `qa.md` is expected and committed; it mutating any
  *other* tracked file is out of scope and must be reverted.

## Verification-first breakdown (TDD adapted: the AC are the tests)

This slice produces prose, so the "tests" are the issue's mechanical acceptance
checks, defined up front and run as the gate:

1. **Red (baseline):** record current state — 8 files < 600 words, missing the
   closing sections. (Captured above.)
2. **Green (deepen):** rewrite each of the 8 to its content contract. Fan out one
   `qa` subagent per file (parallel — distinct files, no conflict), each reading the
   exemplar + `AGENTS.md` + its cross-referenced siblings before writing.
3. **Per-file verify:** an independent `qa` reviewer scores each rewrite against the
   rubric (>=600 words, required sections, >=1 worked example, named tools, no
   duplication, links resolve) and fixes gaps in place.
4. **Consolidate:** one critic reads all 8 + the deep exemplars; flags/merges any
   duplicated concept, contradiction, or broken cross-reference across the set.
5. **Tooling gate:** `uv run python scripts/document-skills.py` exits 0 and renders
   Active Skills cleanly; `uv run python -m solomon_harness.cli compile` does not
   mutate tracked source beyond `qa.md`; `git diff --stat` stays inside the fence.

## Verification criteria (objectively checkable)

- [ ] All 8 target files >= ~600 words, with `## Common pitfalls` and `## Definition of done`.
- [ ] Each names concrete tools/standards and carries >= 1 worked example.
- [ ] No two qa skills cover the same concept; every backticked cross-reference resolves.
- [ ] `document-skills.py` exits 0; Active Skills renders cleanly.
- [ ] `compile` mutates only `agents/qa/agents/qa.md` (the regenerated block), nothing else tracked.
- [ ] `git diff --stat` is confined to the target-files fence (+ `qa.md`).
